1. Промт-инжиниринг и управление контекстом
Это основа экономии токенов: фокус на краткости, релевантности и повторном использовании. Claude предпочитает структурированные промты с XML-like тегами для стабильности, Gemini лучше работает с цепочками мыслей (chain-of-thought), но требует меньше контекста благодаря большему окну (до 1M токенов в моделях 2025+). GPT/Claude имеют лимиты ~128K-200K, так что оптимизация критична для solo-разработчиков с лимитами, а в командах — для снижения общих расходов.
Шаблоны коротких, высокоэффективных промтов

Рекомендация: Используйте императивные глаголы, указывайте точный вывод (e.g., "Верни только код"). Избегайте ненужных деталей; фокусируйтесь на задаче + минимальном контексте.
Анти-паттерн: Длинные описания проблемы без фокуса — тратит токены на "болтовню".
Пример шаблона: "Исправь ошибку в функции [имя]: [краткий diff]. Вывод: только обновленный код."

Системные промты, guardrails, role-based инструкции

Рекомендация: Задайте роль в системном промте (e.g., "Ты — лаконичный Python-эксперт. Отвечай только кодом или инструкциями, без объяснений."). Guardrails: "Не превышай 500 токенов в ответе; если нужно, запроси уточнения."
Различия моделей: Claude хорошо следует ролям с тегами <thinking></thinking>; Gemini — с JSON-структурами для вывода. В командах: унифицируйте системные промты в shared шаблонах.
Анти-паттерн: Отсутствие ролей — модель генерирует verbose ответы, +20-50% токенов.

Техники сокращения и стабилизации контекста

Рекомендация: Используйте summaries: "Вот summary предыдущего контекста: [краткий обзор]. Теперь задача: [новая]."
Стабилизация: Фиксируйте ключевые факты в embeddings (см. раздел 4) или повторяйте только deltas.
Анти-паттерн: Полная пересылка истории — удваивает токены в длинных сессиях.

Повторное использование контекста без его повторной пересылки

Рекомендация: В Claude/Gemini используйте prompt caching (если доступно в API 2025+): кэшируйте системный промт или базовый контекст. В IDE: ссылки на предыдущие ответы (e.g., "Продолжи с ответа #3").
Для команд: Shared memory в инструментах вроде LangChain для агентов.

Стратегии диалогов с минимальной историей

Рекомендация: Stateless подход: каждый промт самодостаточен. Для цепочек: "На основе [краткий summary прошлого]: [новая задача]."
Анти-паттерн: Длинные чаты без очистки — контекст растет экспоненциально.

2. Настройки IDE и LLM-клиентов (Cursor, Antigravity, VS Code, JetBrains)
Cursor (VS Code-based) и Antigravity (Google's agentic IDE на Gemini) оптимизированы для AI; JetBrains/VS Code требуют плагинов (e.g., GitHub Copilot, Claude Code). Solo-разработчики: настройте на минимальный scope; команды: централизованные configs для uniformity.
Параметры, влияющие на объём отправляемого контекста

Рекомендация: В Cursor: Установите "context window limit" в settings на 32K; отключите auto-include unrelated files. В Antigravity: Agent modes с "minimal context" — агенты сами запрашивают данные.
JetBrains/VS Code: В AI Assistant — настройте "token budget" на 10K per query; используйте selective context.
Анти-паттерн: Default full-repo context — тратит 100K+ на простой fix.

Управление автоподгрузкой файлов и scope

Рекомендация: В Cursor/Antigravity: Включите "file scope: current + dependencies only". В VS Code: Плагин Continue.dev с "slash commands" для manual include (e.g., /file filename.py).
Для команд: Git hooks для tagging файлов как "AI-relevant".

Diff / patch / instruction-only режимы вместо передачи полного кода

Рекомендация: В Cursor: Используйте "Apply diff" mode — отправляйте только изменения. В Antigravity: Agents работают с patches natively, снижая токены на 70%.
Анти-паттерн: Полный код в промте — вместо этого: "Применить patch: [diff] к [signature]."

Агентные режимы и их стоимость

Рекомендация: В Antigravity: "Light agent" для простых задач (меньше токенов чем full autonomous). В Cursor: "Composer" mode для multi-step, но лимитируйте steps до 3.
Различия: Claude agents (via Claude Code) дешевле для reasoning-heavy; Gemini в Antigravity — для visual/code gen, но жрет токены на planning.

3. Работа с кодом и изменениями
Фокус на локализации: минимизируйте передачу кода, используйте описания. Для solo: быстрые итерации; команды: стандартизированные шаблоны задач.
Как формулировать задачи без вставки целых файлов

Рекомендация: "Добавь функцию [signature] в модуль [name]: [требования]. Используй стиль [example snippet]."
Анти-паттерн: Вставка всего файла — вместо: "Контекст: class User {...}; задача: добавить метод login."

Локализация изменений (функции, блоки, сигнатуры)

Рекомендация: Выделяйте: "Изменить блок lines 10-20: [diff/instruction]." В Gemini: Используйте "focus on: [code block]".
